数据计算准确性分析报告

Token 计数 · 请求统计 · Cost 估算 — 单模型与聚合计算的正确性验证

一、数据来源与计算链路

每次请求经过代理时，ocgt 记录以下数据链路：

上游响应 (JSON/SSE)
↓ 提取 prompt_tokens / completion_tokens / total_tokens
tokenUsage 结构体 (converter.go:219)
↓ fallback 修正 (cjk estimate / 减法推定)
requestLogEntry (types.go:21)
↓ 写入内存 history[100] + JSONL 日志文件
前端渲染 (app.js updateTrafficStats)
↓ 求和 / 求平均值 / 比例条
用户看到的数据

关键：上游响应中不包含 monetary cost（花费金额）。任何 Cost 展示都只能是基于模型定价表的估算。

二、逐字段准确性分析

字段获取方式可信度说明
input_tokens 从 upstream 响应提取 ✅ 准确 非流式取 Usage.PromptTokens，流式取 SSE chunk.Usage.PromptTokens，Messages 端点解析 usage.input_tokens
output_tokens 从 upstream 响应提取 ✅ 准确 同上，取 CompletionTokens / usage.output_tokens
total_tokens input + output + cache ✅ 准确 就是三个字段的求和，数据源头准确则结果准确
cache_creation_tokens 从 Messages 端点提取 ⚠️ 部分 OpenAI 协议始终为 0（架构限制），仅 Anthropic Messages 端点有值
cache_read_tokens 从 Messages 端点提取 ⚠️ 部分 同上
请求总数 log entries 计数 ✅ 准确 内存 100 条 = 最近 N 条请求的精确计数；JSONL = 时间范围内的精确计数
成功率 status < 300 比例 ✅ 准确 精确计数
平均延迟 Duration 求和 / 请求数 ✅ 准确 精确计算
Cost (花费) — ❌ 未实现 上游不返回花费，ocgt 目前未做估算

三、当前展示存在的问题

⚠️ 问题 1：Token 比例条缺少 Cache 分段
updateTrafficStats 中，分母 totalTokens 包含 cache，但比例条只渲染 input 和 output 两段。当 cache > 0 时，input% + output% < 100%，留下空白段——用户不知道空白代表什么。
input=100, output=100, cache=50 → total=250
input% = 100/250 = 40%
output% = 100/250 = 40%
████████████░░░░░░░░░░░░ ← 20% 空白 = Cache，但无颜色标注
⚠️ 问题 2：无按模型的分组统计
当前所有 token 数据是整个 history 的扁平求和，没有区分模型。deepseek-v4-pro 和 kimi-k2.6 的 token 数混在一起。不同的模型价格不同（deepseek 约 $0.5/M tokens，kimi 约 $1.2/M tokens），混合求和后 Cost 估算会不准。
当前: totalTokens = 2,847,350 (不分模型)
理想: deepseek-v4-pro: 1,520,000 tokens → 估算 $0.76
          kimi-k2.6: 890,000 tokens → 估算 $1.07
          合计: 2,410,000 tokens → 估算 $1.83
❌ 问题 3：无 Cost 估算
完全没有花费展示。Token 数据已经有了，但用户看不到费用信息。对于使用按量计费 (pay-as-you-go) 的用户来说，这是关键信息。
⚠️ 问题 4：前端仅用内存 100 条，不读 JSONL
updateTrafficStats 只计算内存 history[100] 的数据。JSONL 日志文件里存了全部历史，但没有 API 去读。这意味着用户看到的 Token 总和可能严重偏低（流量大时 100 条很快就被覆盖了）。
✅ 没有问题的字段：
input_tokens、output_tokens、请求总数、成功率、平均延迟——这些计算逻辑正确，数据源准确，展示清晰。

四、Cost 估算方案

上游 API 不返回花费金额，但可以基于 token 数和模型定价表来估算。

模型定价参考表

模型Input $/1M tokensOutput $/1M tokens数据来源
deepseek-v4-pro$0.50$1.00官方定价
deepseek-v4-flash$0.10$0.20官方定价
kimi-k2.6$1.20$2.40参考定价
qwen3.6-plus$0.80$1.60参考定价
glm-5.1$0.60$1.20参考定价
其他/未知模型$1.00$2.00保守默认值

计算方式

cost = (input_tokens / 1,000,000) × input_price + (output_tokens / 1,000,000) × output_price

需要 按模型分别计算再求和，不能把所有模型的 token 混在一起算，否则价格差异会导致误差。

展示建议：
- 明确标注 "估算" 或 "≈"
- 显示估算区间：按模型分别计算，然后求和
- 鼠标悬停显示各模型的花费明细

五、单模型 vs 多模型：正确的计算逻辑

场景做法当前状态
请求数 直接计数，不分模型 ✅ 正确
Token 总计 所有模型的 Input + Output + Cache 全量求和 ✅ 正确
Token 按模型 按 model 分组后分别求和 ❌ 未实现
成功率 status 2xx / 总数，不分模型 ✅ 正确
平均延迟 总耗时 / 请求数 ✅ 正确
Cost 估算 按模型分别算再求和 —— 因为各模型单价不同 ❌ 未实现
Cache 占比 cache / total_tokens，按模型展示更有意义 ⚠️ 有数据但无展示

总结

✅ 计算准确的部分
  • input_tokens / output_tokens （依赖上游准确）
  • 请求总数、成功率、平均延迟
  • total_tokens 求和
⚠️ 需要改进的部分
  • 按模型分组统计（缺失）
  • Cost 估算（未实现）
  • Cache 占比展示（比例条缺少一段）
  • JSONL 日志读取（仅用内存 100 条）
